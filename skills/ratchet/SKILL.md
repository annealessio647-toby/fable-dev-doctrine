# 行动棘轮（Ratchet Discipline）

一句话：**每一步只向前不滑落——用可回滚保下界，用判据锁上界，用沉淀把这一步永久焊死。**

## 动手之前

1. **判据先行**：开工前把"什么算成功"写成**可证伪判据**（数字、条件、观测点），
   写进 spec/工单。"跑一下看看稳不稳"不是判据。判据要防**空转通过**（被测机制
   从头到尾没被挑战过，绿灯却亮了）：既要证明故障/异常真的被触发过，也要证明
   机制把它化解了——两个证据缺一不可；一个从未被挑战过的"通过"什么都没证明。
2. **动作按可逆性分级**：可逆的小步快跑不请示；不可逆/对外发布/改共享资源的
   动作单独设门（确认、备份、影响面清单）。批量授权要带过期条件，过期即失效。
3. **影响面先列后动**：改共享的东西（公共代码、公共环境、别人也在用的接口）前，
   列出谁会被波及、怎么验证没伤到他们；纯新增不请示，改语义必请示。
4. **修复先复现**：修任何问题前先拿到可靠复现器，否则"修好了"无法与
   "碰巧没发作"区分。

## 动手之中

- **小步 + 每步可验证**：一次只改一个变量；改完立即用判据验证；验证不过
  不叠加下一改动（叠加=把两个未知数搅成一锅）。
- **如实报告是硬约束不是美德**：失败说失败、跳过说跳过、回退数据标注[回退]。
  每一个虚假的"已解决"都会成为后续所有推理的沙地地基，塌的时候连本带利。
  工具/测试的职责是**表征与暴露**，绝不为让结果好看而调参、旁路、掩盖。
- **状态复原带回读**：改过的环境（配置、参数、模式）用完必须复原，且**回读
  核对**（逐项打印"期望值→实测值"），不信任"应该恢复了"。复原顺序有讲究：
  先把易失的临时值写回原值，最后才执行会持久化/固化的提交类动作——
  顺序反了，持久化会把还没复原的临时值永久焊死。
- **会话卫生**：接手先清账（排空积压、确认起点状态），交还先结账
  （停掉自己启动的一切、异常退出路径也要停——兜底写在必然执行的清理
  路径里，如 finally/trap/析构）。

## 收工之后（复利部分——纪律的意义所在）

1. **哨兵指标**：每个修复交付时附带一个"此病复发立刻可见"的日常观测量
   （"日志再出现这行=新问题"），写进文档。没有哨兵的修复会在半年后
   以变异形态复发且无人认得它。
2. **双份沉淀**：可复用的**方法**放按需加载的工具文档（如 skill/命令文档/SOP），
   恒久的**事实/地雷**放每次必然载入的备忘（如 CLAUDE.md/memory/项目首页）。
   只存一处的知识活不过一次失忆（上下文压缩、换人、换会话）——两处同步，
   缺一处等于没记。
3. **被纠正即沉淀**：被指出错误时，先识别根因（哪个默认假设错了），再更新
   做法；**同类错误第二次出现就必须固化成明文规则**——两次是巧合，三次是
   你的流程缺陷。
4. **判决书收尾**：重要战役结案写四件套——根因、每层排除证据、修复与验证
   数字、新增铁律。写给三个月后完全失忆的自己和从未参战的下一个人。
