# 子代理流水线编排（Subagent Pipeline）

来源：H753 战役中 ARQ 可靠性层（4 任务+终审，抓出 1 个僵死级阻断雷）与 GUI 无线支持
（3 任务）两条流水线的实战沉淀。与 superpowers 的 subagent-driven-development 配套使用，
本 skill 记录的是在其之上验证有效的增强模式。

## 分工模型

- **控制器（强模型）**：写 spec/plan、派发、裁决审查发现、跨任务传递义务、做最终判决。
  控制器**不读大文件全文**——摘要与文件路径经手，细节留给子代理。
- **实现者（执行模型）**：一次一个任务，拿到的是"任务简报+接口契约+全局约束"，
  不是整个 plan（防上下文污染与越权发挥）。
- **审查者（独立子代理）**：实现者的报告只是主张，diff 才是证据。审查者必须亲自
  重跑测试、亲自验证声明（本役审查者用独立 CRC 算法交叉验证 golden 向量、
  用 git show 逐字节比对 ISR、亲自重编译验 md5）。

## 关键模式

1. **义务传递（obligation forwarding）**：任务 N 的审查发现若涉及任务 N+1，
   写成**硬性义务**放进 N+1 的派发词（"T2 必须消费 poll() 的 data 动作，否则洞后
   字节静默丢失"），并要求 N+2 的测试断言它。义务不传递=审查白做。
2. **终审全分支（最强模型）**：逐任务审查只看切片；终审专打**跨任务集成缝隙**
   （生命周期洞、双端语义对齐、"每个切片都对但合起来错"）。本役终审在四个切片
   审查全绿后仍抓出 tx_idle×裸自旋的组合僵死雷——切片审查结构性看不见它。
3. **计划自相矛盾即时勘正**：实现者遵循了约束表而审查者发现 plan 散文与约束表
   冲突时，控制器（=plan 作者）当场改 plan 并注明裁决依据，不让下一个实现者再踩。
4. **台账（progress ledger）**：每任务完成即追加一行到磁盘台账（commit 号+审查结论+
   遗留 Minor）。上下文压缩后台账+git log 是唯一真相，禁止凭记忆重派已完成任务。
5. **Minor 不丢弃**：逐任务审查的 Minor 全部记台账，终审统一 triage
   （必修/后修/接受），关单时逐条销账。
6. **一次一个实现者**：文件有交集的任务严格串行；确无交集（不同仓库/语言）
   且契约冻结在 spec 上时才可并行，风险由后续 golden/一致性任务兜底。

## 派发词要素（实现者）

任务定位一句话 + 简报文件路径（"这是你的需求原文，数值逐字用"）+ 前序任务交付的
接口契约 + 全局约束逐条 + 环境坑（本项目：PowerShell 5.1 无 &&、Read hook 只回首行
需 offset 重读、测试跑法与基线数字）+ 报告文件路径与状态协议
（DONE/DONE_WITH_CONCERNS/NEEDS_CONTEXT/BLOCKED）+ 自审要求。

## 审查词要素

按优先级排序的**对抗焦点清单**（点名 bug 类别让审查者狩猎，而不是"看看有没有问题"）+
必须亲自重跑的命令 + 需要裁决的实现者自曝关注点（禁止预判"这不算问题"——让审查者
独立裁决）+ 输出格式（规格符合性逐条 ✅/❌ + 质量分级 + 裁决 + 亲测结果）。
