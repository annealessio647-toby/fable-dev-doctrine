# 台架纪律（Bench Discipline）

来源：H753 编码器工装战役。一天内十次以上电机启停、四次固件烧录、两条流水线交付，
零安全事故、零数据事故的操作纪律沉淀。

## 安全铁律

1. **电机停机双验**：每个转动过的会话结束时，显式发停速+断使能，然后**位置回读双采样**
   （间隔 1s 的编码器计数差 ≈0 才算静止）——不信任脚本 finally 的自觉，用物理量证明。
2. **每个探针脚本的 finally 都带停机兜底**（停速→断使能→停流），异常路径也要停。
3. **状态复原三件套**：控制模式、刚性/参数快照、流状态——测试改过的都要复原并回读
   核对（"want -> got [OK]"逐项打印）。复原顺序有讲究：易失写回在前、会固化的提交在后。
4. **烧录永远弹窗确认**（选哪个 hex、烧到哪块板），烧前验 MCU device ID，探针唯一性检查。
   批量迭代期的免问授权明确限定轮次，过期即失效。

## 零触碰证明（改共享固件时）

- 高危区（采样 ISR 链等）声明"零触碰"必须**可机器验证**：目标文件 md5 前后逐位一致
  （最强，源码级不变）或反汇编逐条指令 diff（次强，允许重定位地址漂移）。
- "我没改那个文件"不是证明；构建系统的连锁反应（头文件、对齐、字面池）会背叛你。
- 每次重编译后，调试用的内存地址一律从新 map 重取——旧地址读到的是幻觉。

## 验收判据先行

1. **动手前把 PASS 写成可证伪判据**并写进 spec（"soak 10 分钟 rx_bad>0 且 holes==0"
   比"跑一下看看稳不稳"强一个文明等级）。判据要防"空转通过"：证明机制生效的正例
   （坏帧确实发生过）与证明兜底成功的反例（全部被救回）缺一不可。
2. **哨兵指标**：修复交付时附带一个"此病复发即报警"的日常可观测量，写进文档与 memory。
3. **验收数据如实报**：判据没过就是没过，fallback 采到的数据标注 [回退]；
   工具的职责是表征与暴露，不为让某台设备"过"而调参掩盖。
4. **接线逐根对照接线文档验收**——信号线会"看起来能用"，地线缺失是定时炸弹
   （实案：文档写了 common ground required，实物只飞了两根信号线，整个项目的
   "偶发 EMI"全是这条幽灵地环的戏）。

## 双份沉淀（防上下文压缩失忆）

- 可复用方法 → skill/命令文档（按需加载）；恒久事实/地雷 → memory（每会话强制载入）。
  **两处同步，缺一处等于没记。**
- 判决书式记录：根因 + 每层排除证据 + 修复 commit + 验证数字 + 哨兵指标 + 教训清单。
  写给三个月后完全失忆的自己。
- 台账（progress ledger）与文档提交进 git；memory 索引一行一钩子，正文放条目文件。

## 跨会话卫生

- 测试脚本生命周期：开工先"停流+循环排空直到只剩心跳"（单次静默窗判据会被积压中的
  暂停骗过），收工 finally 停流。存储转发型中继（如 ESP 环）会把上一场的尸体倒给
  下一场——会话边界必须主动清账。
- 中断/重启后：先读台账和 git log 再行动，不重做已完成的任务，不信记忆信磁盘。
