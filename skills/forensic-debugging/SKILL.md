# 取证式排除法（Forensic Debugging）

来源：H753 编码器工装 WiFi 链路战役（2026-07）。一个"3 秒超时"表象先后被误判为
环境拥挤、串扰、EMI、Modbus 慢、固件窗口 bug，最终真凶是 PC 读取器 46 行里的行首匹配缺陷。
十几层嫌疑全部靠实验洗白，没有一层靠"应该没问题"放行。

## 铁律

1. **画像不是判决。** 症状与某已知故障相似 ≠ 同一根因。每个嫌疑层必须有专属实验，
   要么定罪要么留下带证据的清白记录（记录本身是资产：下次这层直接跳过）。
2. **一个实验只判一对假设。** 设计判别器（discriminator）：两个候选层在该实验下
   给出**相反**的预期读数。判别器优先用系统里现成的开关（本役用固件既有的 MA0
   停轮询命令区分"ISR 饿死 vs 电气"，零改动零重烧）。
3. **计数器为零先审计计数器。** "诊断显示无异常"要先证明诊断在当前负载下可信
   （本役 ESP 事件队列 32 深被 5000 事件/s 灌爆，溢出证据被静默丢弃，计数器
   高负载下集体撒谎）。同理：自己的显示/日志有没有截断盲区（聚合读只打印头 16 字节
   差点把"回复其实到了"永久掩埋——**全文搜索原始字节日志**才翻案）。
4. **数据前后不一致，先问环境再查代码。** 同一系统昨天好今天坏、且期间零代码变更
   ⇒ 物理层优先（本役=GND 飞线根本没接，接线文档写了实物没插；双向同时变脏是共模/
   共地问题的指纹）。问一句"最近动过什么"比三小时理论便宜。
5. **反直觉数据是金矿，不是噪声。** "忙时反而更快"直接指认了空闲路径的睡眠量子；
   "std 恒等于转速的 3.6%"指认口径问题而非物理噪声（物理噪声不会恰好正比于工作点）。
6. **重试 100% 成功 = 读取端缺陷的经典指纹。** 数据一直都在，是读的人不认识它。
   先查读取器，再查发送端。
7. **跨介质移植审隐性时序。** µs 级介质（USB）上的竞速侥幸（"50ms 排空肯定够"）
   在 ms 级介质（WiFi/TCP）上变成必然失败。列出所有"依赖时序运气"的假设逐个审。

## 流程

1. **全栈嫌疑清单**：画出数据/控制流经过的每一层，每层写出它"作案"时的专属指纹
   （会动哪个计数器、什么时间相关性、什么量纲特征）。
2. **廉价判别器优先**：现成开关 > 改参数 > 加探针 > 改代码重烧。恒定字节率下改突发
   粒度、ICMP 旁路对照（绕过自有栈直测底层）、换命令类型（带副作用 vs 无副作用）
   都是零成本判别器。
3. **现场抓捕**：可复现后，拉长故障窗口（放大超时），在窗口内用带外通道读内部状态
   （SWD/JTAG 热插拔读内存——地址每次重编译后从 map 重取）。
4. **逐位对质**：最终定罪要到字节级——原始记录全文搜索、发送内容与接收内容逐位比、
   状态机变量现场值 vs 预期值。
5. **判决书落盘**：根因 + 排除记录 + 修复验证 + **哨兵指标**（一个"此病复发即报警"的
   可观测量，如"日志再现 retry 行=新问题"）写进项目文档与 memory。

## 反模式

- 修了就跑没有回归判据（修复必须让原故障的复现实验从 100% 变 0%，且全家桶测试零回归）。
- 在"最可疑"的层反复加固而不排除其它层（本役 ARQ 层被冤枉三次，每次都靠计数器自证清白）。
- 把偶发当随机：偶发≠随机，先找触发条件的公共因子（本役全部 stall 都在"采集停止后第一条命令"）。
